📡 1. Project Overview

A telecom-grade SNMP trap monitoring platform for simulated network elements.

The platform consists of two independent Java applications that communicate over the SNMP protocol:

Emulator Telecom Node Emulator
Simulates telecom network elements (BTS, BSC, MSC, etc.) and generates SNMPv2c trap messages for predefined alarm types. Offers both a JavaFX GUI and a CLI interface.
com.snmp.emulator
Server SNMP Manager Server
Listens for incoming SNMP traps on UDP port 162, parses the structured payload, identifies or auto-registers the source node, and persists alarm history into PostgreSQL.
com.snmp.manager

🌲 3. Enterprise OID Tree

Custom MIB structure under Private Enterprise 1.3.6.1.4.1.99999. Follows industry convention: .0.x for notifications, .1.x for data objects.

── iso.org.dod.internet.private.enterprises.99999 ── 1.3.6.1.4.1.99999 ← Enterprise Root │ ├── .0.x Notifications (Trap OIDs — "what happened") │ ├── .0.1 Disk Full │ ├── .0.2 Power Failure │ ├── .0.3 Link Down │ ├── .0.4 Congestion │ ├── .0.5 High Temperature │ ├── .0.6 Memory Exhaustion │ └── .0.7 Configuration Error │ └── .1.x Objects (Variable Bindings — "context/payload") ├── .1.1 nodeName e.g., "Cairo_BTS_01" ├── .1.2 nodeType e.g., "BTS" ├── .1.3 details e.g., "/dev/sda1 at 98%" (optional) └── .1.4 nodeIp e.g., "10.0.0.5" (simulated source IP)

⚡ 4. End-to-End Trap Flow

What happens when the user sends a trap from the emulator.

Step 1
User Input
Select node, alarm type, and optional details
→
Step 2
Build PDU
SnmpService constructs SNMPv2c trap with OID bindings
→
Step 3
UDP Send
Trap sent to target IP on port 162
→
Step 4
Receive & Parse
TrapReceiver ➜ TrapParser extracts structured fields
→
Step 5
Process & Persist
Node lookup/register, severity resolve, save to DB

PDU Structure (on the wire)

Binding OID Example Value
sysUpTime 1.3.6.1.2.1.1.3.0 5000 (timeticks)
snmpTrapOID 1.3.6.1.6.3.1.1.4.1.0 1.3.6.1.4.1.99999.0.1 (Disk Full)
nodeName 1.3.6.1.4.1.99999.1.1 Cairo_BTS_01
nodeType 1.3.6.1.4.1.99999.1.2 BTS
details 1.3.6.1.4.1.99999.1.3 /dev/sda1 at 98% (optional)
nodeIp 1.3.6.1.4.1.99999.1.4 10.0.0.5

Server Processing Logic (TrapService)

  1. Extract IP — Strip port from peer address (e.g., 127.0.0.1/54321 → 127.0.0.1)
  2. Node Lookup — Query nodes table by IP address using nodeIp from the trap (or network source IP if missing)
  3. Auto-Register — If node not found, create it using nodeName, nodeIp, and nodeType from the trap
  4. Action Lookup — Match snmpTrapOID against trap_actions.trap_oid to find severity & action
  5. Persist History — INSERT into trap_history with OID, message (action name + details), status OPEN
  6. Update Node Status — CRITICAL → DOWN, MAJOR/MINOR → WARNING, INFO → UP

🖧 6. Server Component

Package com.snmp.manager — receives, parses, and persists SNMP traps.

Layered Architecture

    TrapReceiver --> TrapParser : parse raw PDU
    TrapParser --> TrapEvent : creates
    TrapReceiver --> TrapService : via TrapListener
    TrapService --> NodeService
    TrapService --> TrapActionDAO
    TrapService --> TrapHistoryDAO
    NodeService --> NodeDAO

Key Features

  • Structured Parsing — Extracts nodeName, nodeType, details from known enterprise sub-OIDs
  • Auto-Registration — Unknown nodes are automatically registered using trap payload data
  • Severity Resolution — Maps trap OID → trap_actions → severity → node status
  • Trap History — Every received trap is persisted with full audit trail
